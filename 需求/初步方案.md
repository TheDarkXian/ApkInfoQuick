2. 功能需求
2.1 前端 - React (GUI)
拖放功能：
使用 react-dropzone 或其他类似的库来实现文件的拖放功能。
拖入 APK 或 AAB 文件后，前端通过 Tauri 调用后端 Rust 进行解析。
信息展示：
显示 APK 或 AAB 的基本信息：包名、产品名、应用名、图标、SDK 版本、API 等级、代码等级、版本号、权限列表、签名信息、ABI、渠道信息等。
其中建议字段定义如下：
- API 等级：`minSdkVersion`、`targetSdkVersion`（可选补充 `compileSdkVersion`，若可解析）。
- 代码等级：`versionCode`（Android 内部版本代码，用于升级比较）。
- 版本号：`versionName`（面向用户展示的版本字符串）。
- 权限列表：`permissions[]`（如 `android.permission.INTERNET`）。
- 签名信息：`signers[]`（每个签名项包含 `scheme`、`certSha256`、`issuer`、`subject`、`validFrom`、`validTo`）。
- ABI：`abis[]`（如 `arm64-v8a`、`armeabi-v7a`、`x86_64`）。
- 渠道信息：`channel`（可来自 manifest `meta-data`、渠道文件或约定字段）。
可以使用 Material-UI 或 Ant Design 提供的组件来构建漂亮的界面。
图标显示：
提取并显示应用图标（PNG 格式）。
使用 <img> 标签显示图标。
按钮与功能：
提供“导出图标”按钮，让用户下载应用图标。
提供按钮查看权限或签名信息等。
提供“查看 ABI / 渠道信息”入口（可折叠面板或详情弹窗）。
2.2 后端 - Rust (解析 APK/AAB)
APK/AAB 解析器：
使用 Rust 库来解析 APK 文件的内容。
可以使用 apk-parser
 或自定义解析器。
提取 APK/AAB 中的 AndroidManifest.xml，获取包名、应用名称、版本、SDK 版本、API 等级、代码等级、版本号、权限列表、ABI、渠道信息等信息。
签名信息通过 APK Signing Block 或证书链解析得到，并统一映射到 `signers` 字段（支持多证书）。
提取图标：
解析 AndroidManifest.xml 和 res 目录中的图标资源。
使用 Rust 库如 image 或 rust-skia 来提取和处理图标。
命令行接口（CLI）：
通过 clap 或 structopt 库实现命令行界面。
支持通过命令行传递文件路径，解析并返回基本信息（JSON 格式），至少包含：
`packageName`、`appName`、`iconUrl`、`minSdkVersion`、`targetSdkVersion`、`versionCode`、`versionName`、`permissions`、`signers`、`abis`、`channel`。
与前端的通信：
使用 Tauri 的 API，在后端处理完成后，将数据传递给前端进行展示。
2.3 Tauri (Bridge)
Tauri API 调用：
前端通过 Tauri 提供的 API 向后端发送请求（通过 invoke）。
后端 Rust 接收到文件后，解析并返回结果（如 JSON 格式）。
前后端通信：
Tauri 作为桥梁，将前端和后端进行连接，确保 React 和 Rust 的交互流畅。

字段缺失策略（必须统一）：
- `versionName` 缺失时返回 `null`，不得伪造默认版本号。
- `compileSdkVersion` 无法解析时返回 `null`。
- `channel` 无法识别时返回 `"unknown"`。
- `permissions`、`abis`、`signers` 无数据时返回空数组 `[]`。
2.5 渠道解析优先级规则
`channel` 的解析来源按以下顺序，命中即停止：
1) `AndroidManifest.xml` 的约定 `meta-data`（如 `UMENG_CHANNEL`、`CHANNEL`）。
2) APK 中约定渠道文件（如 `META-INF/channel_*`）。
3) 文件名规则匹配（如 `app-huawei-release.apk`）。
4) 以上均未命中则返回 `"unknown"`，并在 `warnings` 中写入 `"CHANNEL_NOT_FOUND"`。
2.6 版本范围与 MVP
为了控制首版复杂度，按以下范围执行：
1) `v1.0`：仅支持 APK 解析与展示，完整覆盖本方案定义字段。
2) `v1.1`：新增 AAB 解析，字段契约保持不变。
3) 任何新字段扩展优先放到 `data` 内，避免破坏外层 `Envelope`。
2.7 非功能验收标准
1) 性能：20MB 以内 APK，单文件解析 P95 <= 1.5 秒（桌面常规开发机）。
2) 资源：单次解析额外内存峰值 <= 300MB。
3) 稳定性：连续解析 100 个 APK，成功率 >= 99%，进程无崩溃。
4) 一致性：同一 APK 在 GUI 与 CLI 输出字段值一致（除展示格式差异）。
2.8 测试样本矩阵与合规要求
最小测试矩阵：
1) 多 ABI（`arm64-v8a`/`armeabi-v7a`/`x86_64`）。
2) 无图标或图标资源损坏。
3) Manifest 字段缺失（如无 `versionName`）。
4) 损坏 APK / 非法 ZIP / 加固样本。
5) 多签名方案（v1/v2/v3/v4）及多证书。
6) 多渠道包与未知渠道包。
7) 不同 targetSdkVersion（21~34+）样本。
安全与合规：
1) 仅本地解析，不上传 APK/AAB 内容。
2) 日志中禁止输出完整证书原文，仅记录摘要（如 SHA256）和必要错误码。
3) 错误信息避免泄露本机敏感路径，仅保留相对文件名或脱敏路径。
3. 核心流程
拖放文件：用户将 APK 或 AAB 文件拖放到前端界面中。
前端调用后端：前端通过 Tauri 的 invoke 调用后端，传入文件路径。
后端解析文件：Rust 后端接收文件路径，使用 Rust 库解析文件并提取元数据（包名、产品名、图标、API 等级、代码等级、版本号、权限列表、签名信息、ABI、渠道信息等）。
返回数据：后端将解析结果按 `Envelope + data`（JSON 格式）返回给前端。
前端展示：前端接收到数据后，动态渲染页面，显示文件的基本信息。

5.2 Rust 后端 (parser.rs)
use apk_parser::Apk;
use std::fs::File;
use std::path::Path;

#[tauri::command]
fn parse_apk(file_path: String) -> Result<String, String> {
    let path = Path::new(&file_path);
    let apk = Apk::from_path(path).map_err(|e| e.to_string())?;
    let package_name = apk.get_package_name().unwrap_or_default();
    let app_name = apk.get_app_name().unwrap_or_default();
    let icon_url = apk.get_icon_url().unwrap_or_default();
    let min_sdk_version = apk.get_min_sdk_version().unwrap_or_default();
    let target_sdk_version = apk.get_target_sdk_version().unwrap_or_default();
    let version_code = apk.get_version_code().unwrap_or_default();
    let version_name = apk.get_version_name().unwrap_or_default();
    let permissions = apk.get_permissions().unwrap_or_default();
    let abis = apk.get_abis().unwrap_or_default();
    let channel = apk.get_channel().unwrap_or_else(|| "unknown".to_string());
    let signers = apk.get_signature_info().unwrap_or_default();

    Ok(format!(
        "{{ \"packageName\": \"{}\", \"appName\": \"{}\", \"iconUrl\": \"{}\", \"minSdkVersion\": \"{}\", \"targetSdkVersion\": \"{}\", \"versionCode\": \"{}\", \"versionName\": \"{}\", \"permissions\": {:?}, \"signers\": {:?}, \"abis\": {:?}, \"channel\": \"{}\" }}",
        package_name, app_name, icon_url, min_sdk_version, target_sdk_version, version_code, version_name, permissions, signers, abis, channel
    ))
}

- 负责范围：样本矩阵维护、契约测试、回归测试、发布验证。
- 关键输出：测试报告、缺陷单、风险清单。
- 交接标准：关键路径通过，P1/P2 缺陷清零或有明确豁免。
11.2 RACI（简化）
1) 契约定义（Schema）：PM(A) / 后端(R) / 前端(C) / QA(C)
2) APK 字段解析：后端(RA) / QA(C) / PM(I) / 前端(I)
3) GUI 展示与交互：前端(RA) / PM(C) / QA(C) / 后端(I)
4) 测试样本矩阵：QA(RA) / 后端(C) / 前端(C) / PM(I)
5) 版本验收与发布：PM(A) / QA(R) / 后端(C) / 前端(C)
