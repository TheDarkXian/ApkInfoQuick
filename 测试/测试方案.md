# ApkInfoQuick 测试方案（基于当前项目现状）

## 1. 现状评估（2026-04-23）

### 1.1 仓库与实现状态

- 项目结构完整：`backend`（Rust 解析与 CLI）、`frontend`（React + Tauri）。
- 前端已实现：APK 选择/拖拽、调用 `parse_apk`、Envelope 展示、错误态/重试、图标导出、warnings 展示。
- 后端已实现：APK 读取、Manifest 解析（文本 XML + 部分二进制 AXML）、ABI/渠道/图标/signers（best-effort）提取、统一 `Envelope + data` 输出。

### 1.2 已有自动化测试

- 前端：
1. `frontend/src/types/apk.test.ts`（字段归一化与默认值）
2. `frontend/src/services/tauri.test.ts`（Tauri invoke 调用层）
- 后端：
1. `backend/src/parser.rs` 内含单元测试（错误码、默认值、渠道优先级、契约键、脱敏、Tauri 一致性等）
- 当前验证结果：
1. `npm.cmd run test -- --run` 通过（3 tests）。
2. 后端 `cargo test` 当前机器无法执行（`cargo/rustup` 缺失）。

### 1.3 当前主要风险

1. 后端与 Tauri 联调在本机不可执行（环境缺 Rust toolchain）。
2. 现有自动化偏“单元级”，缺端到端（GUI 拖拽到结果展示）自动回归。
3. 缺性能/稳定性压测基线（P95、100包连续解析成功率）。
4. 签名解析为 best-effort，需针对 warnings 与字段完整性做专门策略测试。

## 2. 测试目标

1. 验证 GUI 与 CLI 对同一 APK 输出字段一致。
2. 验证契约 `Envelope + data` 稳定、默认值策略符合需求。
3. 验证关键异常路径（损坏包、非 APK、缺字段、无渠道）可控且可读。
4. 建立可重复执行的回归机制（本地与 CI）。

## 3. 测试范围

### 3.1 本轮范围（v1.0）

- APK 解析与展示（不含 AAB）。
- CLI 输出与 Tauri/GUI 输出一致性。
- 字段与默认值策略、渠道优先级、错误码与脱敏。
- 非功能基线：性能、稳定性、资源占用（按样本可执行）。

### 3.2 暂不覆盖

- AAB 解析（v1.1）。
- 完整证书链语义正确性（当前实现为 best-effort，仅验证“行为可预期 + warnings 完整”）。

## 4. 测试策略

### 4.1 分层策略

1. 单元测试（快速）：解析规则、字段映射、默认值、错误码与脱敏。
2. 契约测试（核心）：Schema/关键字段存在性、类型与缺省规则。
3. 组件与服务测试（前端）：输入校验、状态切换、warnings 呈现。
4. 端到端测试（高价值）：拖拽 APK -> invoke -> 展示/错误重试。
5. 非功能测试（发布门禁）：P95、内存峰值、100包稳定性。

### 4.2 风险驱动优先级

- `P0`：契约一致性、错误码稳定、GUI/CLI 同包一致、解析不崩溃。
- `P1`：图标导出、签名 warning 语义、渠道优先级全覆盖。
- `P2`：UI 细节与可用性增强（长列表交互、搜索/复制体验）。

## 5. 测试设计（按模块）

## 5.1 后端（Rust）

1. 输入校验：不存在路径、非文件、非 `.apk`、损坏 zip。
2. Manifest 解析：
- 文本 XML 正常路径。
- 二进制 AXML 部分能力路径（并校验 `MANIFEST_BINARY_PARTIAL` warning）。
3. 规则校验：
- 默认值：`versionName/compileSdkVersion/channel/数组字段`。
- 渠道优先级：manifest > META-INF/channel_* > 文件名 > unknown + warning。
4. 安全与输出：
- 错误信息路径脱敏。
- signers 字段结构与 `SIGNATURE_PARTIAL` 逻辑一致。

## 5.2 前端（React）

1. 文件输入：
- 仅允许 `.apk`。
- 无 path 场景提示“需 Tauri 桌面模式”。
2. 结果渲染：
- `success=true` 展示所有分区（基础信息/版本/权限/ABI/签名/warnings）。
- `success=false` 展示 `errorCode/errorMessage` 与重试入口。
3. 行为验证：
- 重试逻辑复用 `lastFilePath`。
- 图标导出按钮可用性与禁用态正确。
- `SIGNATURE_PARTIAL` / `SIGNATURE_BLOCK_DETECTED_UNPARSED` 告警条显示正确。

## 5.3 一致性与端到端

1. 同一 APK，CLI 与 GUI 的 `data` 字段对比一致。
2. 端到端脚本覆盖关键路径：
- 正常 APK。
- 缺失字段 APK。
- 损坏 APK。

## 6. 测试环境与前置条件

1. Node/npm/Tauri CLI 已具备（当前机器已满足）。
2. Rust 工具链为阻塞项（当前机器缺 `cargo/rustup`）：
- 安装：`winget install Rustlang.Rustup`
- 重启终端后执行：`npm.cmd run doctor`
3. 建议统一 Windows 环境执行，避免路径与 WebView2 差异。

## 7. 测试数据策略

1. 样本分层：
- `golden`：可稳定解析的标准 APK（含 icon/permissions/signature）。
- `edge`：缺字段、无图标、未知渠道、多 ABI。
- `bad`：损坏 zip、伪 APK、异常输入。
2. 样本管理：
- 建议建立 `tests/fixtures`（或 `测试/fixtures`）并附样本说明清单。
- 每个样本标注“命中规则”和“预期 warnings/errorCode”。

## 8. 执行计划（两阶段）

### 阶段 A：可立即执行（当前环境）

1. 跑前端单测并纳入基线（已通过）。
2. 完善测试文档与样本清单（本方案 + 测试清单）。
3. 新增前端组件测试（拖拽/错误态/warnings/UI 交互）。

### 阶段 B：补齐 Rust 环境后执行

1. 运行 `cargo test` 并补齐缺失用例（边界+错误+警告）。
2. 建立 CLI 契约测试脚本（JSON Schema 校验）。
3. 跑端到端联调（Tauri dev）与 GUI/CLI 一致性回归。
4. 跑非功能基线（P95/内存/100包稳定性），形成报告。

## 9. 质量门禁（发布前）

1. 自动化：
- 前端单测全绿。
- 后端单测全绿。
- 契约校验通过率 100%。
2. 关键业务：
- P0 用例 100% 通过。
- GUI/CLI 同包字段一致。
3. 非功能：
- 20MB APK 解析 P95 <= 1.5s。
- 单次额外内存峰值 <= 300MB。
- 100包连续解析成功率 >= 99%。
