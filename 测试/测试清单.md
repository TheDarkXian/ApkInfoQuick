# ApkInfoQuick 测试清单（v1.0）

## 1. 测试目标与范围

- 目标：验证 APK 解析在 GUI 与 CLI 下行为一致，且满足需求文档中的功能与非功能验收标准。
- 版本范围：`v1.0`（仅 APK）。
- 不在本轮范围：AAB 解析（计划 `v1.1`）。

## 2. 测试分层与优先级

- `P0`：阻塞发布，必须通过。
- `P1`：核心能力，发布前应通过。
- `P2`：增强项，可在迭代中补齐。

## 3. 冒烟测试（P0）

### T-SMOKE-001 APK 基础解析（GUI）

- 优先级：`P0`
- 前置：准备一个正常 APK（包含图标、权限、签名信息）。
- 步骤：
1. 启动应用（Tauri GUI）。
2. 拖拽 APK 到上传区域。
3. 观察基础信息区与详情区展示。
- 预期：
1. 页面成功展示 `packageName`、`appName`、`iconUrl`、`minSdkVersion`、`targetSdkVersion`、`versionCode`、`versionName`、`permissions`、`signers`、`abis`、`channel`。
2. 无崩溃、无卡死。
3. 若有 `warnings`，可见且可读。

### T-SMOKE-002 APK 基础解析（CLI）

- 优先级：`P0`
- 前置：同 T-SMOKE-001 APK。
- 步骤：
1. 执行 `parse` 命令并传入 APK 路径。
2. 获取 JSON 输出。
- 预期：
1. 输出符合 `Envelope + data` 顶层结构。
2. `success=true` 时 `errorCode/errorMessage` 为 `null`。
3. 字段齐全并可被 JSON 解析器正确解析。

### T-SMOKE-003 GUI/CLI 一致性

- 优先级：`P0`
- 前置：同一 APK 分别走 GUI 与 CLI。
- 步骤：
1. 保存 GUI 结果（可通过日志或调试面板）。
2. 保存 CLI JSON 结果。
3. 对比 `data` 字段。
- 预期：
1. 同名字段值一致（允许仅展示格式差异）。
2. 差异项必须可解释且记录在案。

## 4. 契约测试（P0/P1）

### T-CONTRACT-001 Envelope 必备字段校验

- 优先级：`P0`
- 步骤：对任一成功解析结果做 Schema 校验。
- 预期：
1. 顶层必须存在：`success/data/errorCode/errorMessage/warnings`。
2. 禁止未定义额外顶层字段（`additionalProperties=false`）。

### T-CONTRACT-002 data 必备字段校验

- 优先级：`P0`
- 步骤：对 `data` 做字段与类型校验。
- 预期：
1. 必备字段全部存在。
2. 类型正确：例如 `minSdkVersion/targetSdkVersion/versionCode` 为整数，`permissions/signers/abis` 为数组。

### T-CONTRACT-003 缺省策略校验

- 优先级：`P0`
- 样本：字段缺失 APK（例如缺 `versionName`）。
- 预期：
1. `versionName` 缺失返回 `null`。
2. `compileSdkVersion` 无法解析返回 `null`。
3. `channel` 无法识别返回 `"unknown"`。
4. `permissions/abis/signers` 无数据时返回 `[]`。

### T-CONTRACT-004 渠道解析优先级

- 优先级：`P1`
- 样本：分别命中 `manifest meta-data`、`META-INF/channel_*`、文件名规则的 APK。
- 预期：
1. 解析优先级严格为：manifest > channel 文件 > 文件名。
2. 全部未命中时：`channel="unknown"` 且 `warnings` 包含 `"CHANNEL_NOT_FOUND"`。

## 5. 功能回归测试（P1）

### T-FUNC-001 图标展示与导出

- 优先级：`P1`
- 预期：
1. GUI 正常预览图标。
2. 导出图标可打开、格式正确（PNG）。
3. 无图标时展示兜底 UI，不崩溃。

### T-FUNC-002 权限与签名详情可读性

- 优先级：`P1`
- 预期：
1. 可展开查看权限与签名信息。
2. 长列表可滚动，必要时支持搜索/复制（若已实现）。
3. 多签名场景可完整显示 `signers[]`。

### T-FUNC-003 错误态与重试

- 优先级：`P1`
- 样本：损坏 APK、非法 ZIP。
- 预期：
1. 返回稳定错误码与错误信息。
2. GUI 展示 `errorCode/errorMessage/warnings`。
3. 可重试且不会导致应用进入异常状态。

### T-FUNC-004 ABI 与版本信息正确性

- 优先级：`P1`
- 样本：多 ABI 包、不同 targetSdkVersion（21~34+）。
- 预期：
1. `abis[]` 准确反映包内 ABI。
2. `minSdkVersion/targetSdkVersion/versionCode/versionName` 准确。

## 6. 非功能测试（P0/P1）

### T-NFR-001 性能基线

- 优先级：`P0`
- 条件：APK 大小 <= 20MB。
- 步骤：批量运行并统计耗时分位。
- 预期：单文件解析 `P95 <= 1.5s`。

### T-NFR-002 内存峰值

- 优先级：`P1`
- 步骤：解析过程监控额外内存占用。
- 预期：单次解析额外内存峰值 `<= 300MB`。

### T-NFR-003 稳定性（长稳）

- 优先级：`P0`
- 步骤：连续解析 100 个 APK。
- 预期：
1. 成功率 `>= 99%`。
2. 进程无崩溃。

## 7. 安全与合规检查（P0）

### T-SEC-001 本地处理约束

- 优先级：`P0`
- 预期：APK/AAB 内容仅本地处理，不上传远端。

### T-SEC-002 日志脱敏

- 优先级：`P0`
- 预期：
1. 日志不输出完整证书原文。
2. 仅保留摘要（如 SHA256）与必要错误码。

### T-SEC-003 路径脱敏

- 优先级：`P0`
- 预期：错误信息不泄露敏感绝对路径，仅输出脱敏路径或文件名。

## 8. 最小样本矩阵（建议）

1. 多 ABI 样本：`arm64-v8a`/`armeabi-v7a`/`x86_64`。
2. 缺图标或图标损坏样本。
3. Manifest 字段缺失样本（缺 `versionName`）。
4. 损坏 APK/非法 ZIP/加固样本。
5. 多签名样本（v1/v2/v3/v4 + 多证书）。
6. 多渠道与未知渠道样本。
7. targetSdkVersion 覆盖 21~34+ 样本。

## 9. 发布判定（DoD 对齐）

1. 代码与关键流程测试通过（后端单测 + 前端关键流）。
2. 契约校验通过（Schema 100%）。
3. GUI/CLI 同包输出字段一致。
4. 非功能指标达到需求阈值。
5. 文档完整：字段说明、错误码说明、已知限制。
